# Comprehensive Project Overview: Emotion-Aware Adaptive Learning Platform

This document serves as the **ultimate step-by-step technical guide** to the entire architecture, data flow, structure, and execution of the Emotion-Aware Adaptive Learning system. It traces every detail from initial boot up to the deep AI reinforcement learning loops beneath the surface.

##  1. Architecture & Core Tech Stack

The platform is designed around a Streamlit UI communicating with multiple local AI endpoints, a foundational sqlite database, and computer vision libraries. 

- **Frontend & Routing:** Streamlit (`app.py`, `pages/`)
- **Database Layer:** SQLite (`database/db.py`)
- **Computer Vision (Emotion):** OpenCV (`cv2`) and DeepFace (`emotion/face_emotion.py`)
- **NLP Sentiment Analysis:** SpaCy / NLTK / TextBlob (`emotion/text_emotion.py`)
- **LLM Core (Content & Quizzes):** Local Ollama runtime running `qwen2:0.5b` (`adaptive/adaptive_engine.py`).
- **Visual AI Integration:** HuggingFace Stable Diffusion XL API (`emotion/image_generator.py`).
- **Reinforcement Learning Engine:** Custom Q-Learning Python implementation (`adaptive/rl_agent.py`).
- **Audio Output (TTS):** Google Text-to-Speech (gTTS).

---

##  2. Directory & Structure Breakdown Step-By-Step

- **`app.py`**: The main entry point. Sets up root layout, handles User Authentication (Admin/Student Roles), and routes users to the corresponding dashboard.
- **`database/db.py`**: Contains `init_db()` which builds the SQLite architecture (`sessions`, `quiz_history`, `rl_q_table`, `instagram_interests`). It uses schema migrations via `ALTER TABLE` to append new fields dynamically like `is_depressed`.
- **`emotion/` (Sensing & Output)**:
  - `face_emotion.py`: Real-time frame analysis from the webcam.
  - `text_emotion.py`: Extract sentiment and keyword from the user's typed queries.
  - `image_generator.py`: Pings HuggingFace API based on topics and detected interests.
  - `adaptive_agent.py`: Constructs the avatar representations and formats AI prompt templates.
- **`adaptive/` (Logic & Learning Loops)**:
  - `adaptive_engine.py`: Defines how the local Ollama LLM should react. Parses complex Quiz outputs using Zero-Space Tolerant Regex parsing. Connects directly to the RL mechanism (`rl_agent.update`).
  - `rl_agent.py`: A Q-learning table manager. State = (Emotion, Last Score). Action = (Difficulty Level). Reward = Accuracy mapping.
- **`pages/` (Interactive Interfaces)**:
  - `user_dashboard.py`: Welcome page for students. Links to Learning and Quiz.
  - `learning.py`: The live classroom. Tracks faces, records user inputs, calculates question thresholds.
  - `quiz.py`: Automatically triggered every 10 queries. Syncs with RL to provide custom difficutly MCQs.
  - `story.py`: Immersive multimedia page with generated visuals and TTS.
  - `admin_dashboard.py`: A deep metric visualization suite for teachers/admins measuring student mental health risk factors (`is_depressed`).

##  3. Step-by-Step Executable Workflows

### **Workflow A: The Learning & Interaction Loop**
1. **Initiation**: The student logs in and navigates to the `Learning Session` (`pages/learning.py`).
2. **Biometric Tracking**: A webcam component opens using OpenCV. Every frame is passed to `detect_face_emotion_from_frame()`, capturing real-time emotion (e.g., *Neutral*, *Confused*).
3. **User Query**: The student types a question in the chat text box.
4. **Sentiment & Intent Layer**: The backend extracts keywords, measures text sentiment (Positive/Negative), and checks if the student uses depressive keywords (triggering the `is_depressed` flag in DB).
5. **Contextual GenAI Trigger**: The LLM Engine (via Ollama `qwen2:0.5b`) is primed. The system passes: *User Question* + *Detected Emotion* + *Detected Sector/Interest*. 
6. **Delivery**: The system displays the localized LLM response. It can also read this response aloud using `gTTS` or route to `pages/story.py` to fetch an educational illustration from the Hugging Face Stable Diffusion inference node.
7. **Increment**: The system adds to the student's `question_count`. Once this hits exactly **10 questions**, the learning module locks and prompts the student to take a quiz.

### **Workflow B: The Reinforcement Learning (RL) Quiz Loop**
1. **Initialization**: On entering `pages/quiz.py`, the system fetches previous context: the student's **Last Emotion** and **Last Quiz Score**.
2. **State Selection (Q-Learning)**: 
   - `rl_agent.get_state` maps previous inputs to a simplified discrete state (e.g., `Sad_High`, `Neutral_Low`).
   - The Q-Table fetches the most optimal "Action" mapping to the highest Expected reward. Actions are `Easy`, `Medium`, or `Hard` difficulty.
3. **Adaptive Quiz Generation**: The selected difficulty is passed back to `adaptive_engine.generate_single_mcq()`. The Ollama LLM is prompted strictly to return an MCQ matching that difficulty parameter. Deep regex validation ensures formatting.
4. **Scoring & Feedback**: The student answers. A correct answer yields `+1` reward; an incorrect answer yields `-1` penalty (via custom reward balancing: `correct - wrong`).
5. **Q-Table Update**: `agent.update(state, difficulty, reward, next_state)` calculates the temporal difference (TD) target, applies the learning rate (alpha), and updates the SQLite `rl_q_table` DB synchronously.
6. **Result Handling**: The session state is cleared, allowing the student to return to `pages/learning.py` for their next 10 questions.

### **Workflow C: Mental Health & Admin Safety Net**
1. **Ingestion**: As students converse in Workflow A, the `sessions` table tracks the flag `is_depressed`, emotion metadata, and user intent.
2. **Observation**: Teachers log into `app.py` as an Admin, navigating to `pages/admin_dashboard.py`.
3. **Data Distillation**: The backend triggers mathematical algorithms mapping total negative inputs to positive inputs. 
4. **Risk Flagging**: If the ratio of "sad/angry/negative" to "neutral/happy" falls below predefined thresholds, the Admin Dashboard places the student in the **At-Risk category**, plotting metrics via Streamlit graphs.

##  4. Data Layer Architecture (SQLite)

- **`users` Table:** Stores raw credentials and basic RBAC (Role-Based Access Control - `user` vs `admin`).
- **`sessions` Table:** Massively detailed logging array catching `timestamp`, `face_emotion`, `face_confidence`, and the `is_depressed` risk trigger.
- **`quiz_history` Table:** A ledger of exactly what a student scored, on what difficulty, under what emotion.
- **`rl_q_table` Table:** Persistent memory for the Reinforcement Agent. Columns: `state` (Primary Key), `action` (Primary Key), `q_value`. 
- **`instagram_interests` Table:** Tracks personalized hobbies for context mixing parsed from external zip/json imports.

##  5. Deployment & Tooling Setup Instructions

1. **Python Environment:** The project leans entirely on Python 3.9+ within a `venv`. Run `pip install -r requirements.txt`.
2. **Local Model Engine (Crucial):**
   - Download and install Ollama.
   - Execute `ollama pull qwen2:0.5b` (or swap to `phi3` if instructed by the adaptive engine config) to load the language model weights. MUST run in the background for LLM endpoints to stay alive.
3. **External Authentication:**
   - Modify `emotion/image_generator.py` and ensure the `HF_API_TOKEN` is active to get correct stability-ai inference.
4. **Execution:** Use `streamlit run app.py` at the root folder to boot the web application.

##  Summary of the Synergistic Behavior

Every single interaction a student makes is monitored, scored, and passed through parallel logic paths:
* If They Look Sad -> LLM tone shifts to Supportive -> Quiz gets Easier -> System monitors for depression keywords -> Admin gets notified if prolonged.
* If They Ace the Content -> RL tracks 'High Score' State -> Increases Q-Value for 'Hard' action -> Next Quiz incorporates analytical complex LLM generation. 

This constructs a constantly shifting, fully personalized educational loop heavily reliant on local AI inference routing.
